# Patient Agent 面试复习手册

## 1. 项目定位

这个项目的目标是做一个面向患者场景的智能 Agent，核心是把患者基础信息、病例、诊疗记录、记忆能力、多模态能力和工具调用整合到一条可控的问答链路里。

项目重点不只是“大模型回答问题”，而是让 Agent 在真实业务约束下完成：

- 患者身份识别与隐私边界控制
- 病例和就诊记录查询
- 多轮对话连续性
- 长期记忆沉淀
- 图片理解与语音播报
- 工具化执行和规划推理

一句话概括：

> 这是一个面向患者咨询场景的多模态、带记忆、带规划、可扩展的医疗问答 Agent。

---

## 2. 整体架构

系统可以拆成 6 层：

1. API 接入层
2. Agent 执行层
3. Tool / MCP 能力层
4. Memory 记忆层
5. Multi-modal 多模态层
6. 数据存储层

### 2.1 API 接入层

主要入口是：

- `POST /api/agent/query`
- `POST /api/memory/extract/business`
- `POST /api/memory/extract/conversation`
- `POST /api/memory/search/events`
- `GET /api/memory/profile`
- `GET /api/memory/events`
- `PUT /api/memory/preferences`

`/api/agent/query` 是主入口，其他记忆接口负责记忆沉淀、检索和配置。

### 2.2 Agent 执行层

核心类是：

- `QwenMCPAgent`

这一层负责：

- 接收用户问题和图片
- 注入记忆上下文
- 执行 Planner
- 驱动工具调用
- 汇总最终答案

### 2.3 Tool / MCP 能力层

系统把内部能力封装成工具，例如：

- 患者身份验证
- 查询患者资料
- 查询病例
- 查询就诊记录

这些能力通过统一的工具协议被 Agent 调用，本质上是 MCP 风格的工具层。

### 2.4 Memory 记忆层

记忆分两层：

- 短期记忆：当前会话的最近几轮对话
- 长期记忆：用户画像、关键事件、长期偏好

### 2.5 Multi-modal 多模态层

支持两类多模态能力：

- 图片理解
- 语音播报

### 2.6 数据存储层

当前用两种存储：

- SQLite：存结构化业务数据和记忆数据
- FAISS：存关键事件的向量索引

## 3. 记忆架构

对应 API：

- `PUT /api/memory/preferences`
- `GET /api/memory/preferences`
- `GET /api/memory/events`
- `GET /api/memory/profile`
- `POST /api/memory/search/events`
- `POST /api/memory/extract/business`
- `POST /api/memory/extract/conversation`
- `POST /api/memory/conversations`
- `GET /api/memory/conversations`

### 3.1 为什么要分短期记忆和长期记忆

短期记忆和长期记忆的数据来源、生命周期、用途不同，必须分层。

#### 短期记忆

短期记忆是会话态上下文，核心目的是保证多轮对话不断片。

主要保存：

- 最近几轮用户问题
- 最近几轮助手回答
- 当前会话中的多模态摘要

特点：

- 高频写入
- 生命周期短
- 主要服务当前问答上下文

#### 长期记忆

长期记忆是跨会话仍然有效的信息。

主要包含两类：

1. 用户主动配置的长期偏好
2. 系统沉淀出来的长期业务记忆

长期业务记忆又拆成：

- 用户画像
- 关键事件

特点：

- 低频刷新
- 生命周期长
- 主要服务个性化回答和跨会话召回

### 3.2 当前记忆表设计

#### 短期记忆表

`conversation_memories`

保存字段包括：

- `patient_id`
- `session_id`
- `role`
- `content`
- `multimodal_payload`
- `created_at`

这里不再保存 `task_context` 和 `temporary_preference`，因为当前版本把短期记忆收敛成最核心的对话事实。

#### 长期关键事件表

`memory_events`

保存字段包括：

- `patient_id`
- `event_type`
- `event_time`
- `title`
- `summary`
- `source_type`
- `source_id`

#### 用户画像表

`user_profiles`

保存字段包括：

- `profile_summary`
- `communication_style`
- `preferred_topics`
- `stable_preferences`
- `source_summary`
- `refreshed_at`

#### 长期偏好配置表

`memory_preferences`

保存用户主动设置的长期偏好，例如：

- 偏好称呼
- 回答风格
- 回答长度
- 语言
- 关注主题
- 额外偏好

## 4. 短期记忆的计数方案

这是面试里很容易被问到的点。

### 4.1 有没有单独建计数表

没有。

当前没有专门建“轮次计数表”或者“短期记忆计数表”，而是直接基于 `conversation_memories` 做聚合统计。

### 4.2 当前怎么计数

系统在每次问答完成后，会往短期记忆表写两条记录：

- 一条 `role=user`
- 一条 `role=assistant`

所以：

- 1 轮问答 = 2 条短期记忆

计数逻辑就是直接：

- 按 `patient_id` 对 `conversation_memories` 做 `count(*)`

### 4.3 为什么这样设计

优点：

- 简单
- 不需要维护额外状态
- 不容易出现计数漂移
- 对当前体量足够稳定

缺点：

- 如果后期数据量很大，频繁 count 成本会比单独计数器高

但当前这个项目阶段，这是更稳的做法。

### 4.4 5 轮触发长期记忆沉淀

当前的自动触发规则是：

- 每轮问答会写 2 条短期记忆
- 当某个患者短期记忆累计达到 10 条时
- 自动触发一次“从短期对话提炼长期记忆”的过程

也就是：

- 5 轮 = 10 条消息

触发条件是：

- `short_term_count >= 10`
- 并且 `short_term_count % 10 == 0`

这意味着：

- 到第 5 轮触发一次
- 到第 10 轮再触发一次
- 不会每次问答都重复跑抽取

### 4.5 为什么按消息数触发而不是按 session 触发

因为当前 `query` 接口不暴露 `session_id` 给外部控制，短期记忆是服务端自动写入的。

按消息总数触发有两个优势：

- 逻辑稳定，不依赖客户端显式传 session
- 适合“用户自然连续咨询”的场景

面试时可以这样说：

> 我没有单独维护轮次计数器，而是直接对短期记忆表做聚合统计。由于每轮问答固定写两条消息，所以可以稳定映射成 5 轮阈值触发长期记忆沉淀，这样实现简单且不容易发生状态漂移。

## 5. 长期记忆的生成与存储

### 5.1 长期记忆分两条链路生成

对应 API：

- `POST /api/memory/extract/business`
- `POST /api/memory/extract/conversation`

长期记忆抽取被拆成两条链路，而不是一个统一接口。

#### 业务数据抽取

来源：

- `medical_cases`
- `visit_records`

产出：

- 长期关键事件

特点：

- 低频
- 稳定
- 适合增量刷新或定时刷新

#### 短期对话抽取

来源：

- `conversation_memories`

产出：

- 对话类关键事件
- 用户画像

特点：

- 高频
- 偏好和关注点更明显
- 适合按阈值触发沉淀

### 5.2 为什么要拆成两条链路

因为两类数据在：

- 更新频率
- 抽取目标
- 刷新成本
- 业务性质

上都不同。

如果统一抽取，每次都重扫病例和诊疗记录，成本高且没必要。

### 5.3 长期记忆写到哪里

相关 API：

- `GET /api/memory/events`
- `GET /api/memory/profile`
- `PUT /api/memory/preferences`
- `GET /api/memory/preferences`

长期记忆是双写模式。

#### 结构化存储

写入 SQLite：

- `memory_events`
- `user_profiles`
- `memory_preferences`

#### 向量存储

关键事件会额外做向量化：

- 使用 Qwen embedding 模型
- 当前默认 `text-embedding-v4`
- 写入本地 FAISS 索引

注意：

- 当前只对关键事件做向量化
- 用户画像暂时不做向量化
- 这一步不是无条件执行，只有在 `faiss` / `numpy` 可用时才会真正写入 FAISS
- 如果向量依赖不可用，系统仍然会保留结构化长期记忆，但会跳过向量写入

### 5.4 混合检索怎么做

对应 API：

- `POST /api/memory/search/events`

关键事件的检索不是纯向量，也不是纯关键词，而是混合检索。

流程：

1. 先根据查询词做关键词匹配
2. 在向量依赖和 embedding 配置可用时，再做向量相似度检索
3. 对结果做合并和综合打分
4. 返回 topN

返回时会明确标出：

- `keyword`
- `vector`
- `hybrid`
- `recent`

也会返回：

- `keyword_score`
- `vector_score`
- `retrieval_score`

这样可以解释每条记忆为什么被召回。

这里也要说明边界：

- 如果 FAISS 不可用，向量检索会被跳过
- 如果 `QWEN_API_KEY` 没有配置，Qwen embedding 也无法执行
- 在这些情况下，系统会退化成关键词检索或最近事件回退

## 6. MCP / 工具调用能力

### 6.1 这个项目的 MCP 能力是什么

这里的 MCP 不一定是“远程独立进程的标准 MCP Server 全家桶”，而是 MCP 风格的工具化设计。

核心思想是：

- 把内部能力封装成标准工具
- 由 Agent 自动路由调用
- 工具调用结果作为结构化证据返回给 Agent

### 6.2 当前封装的主要工具能力

主要包括：

- 患者身份验证
- 查询患者基础信息
- 查询病例
- 查询就诊记录

### 6.3 为什么说是模块化 MCP Server

因为系统不是把所有逻辑写死在 prompt 里，而是把业务能力按模块拆成工具层，Agent 只负责：

- 理解意图
- 规划动作
- 选择工具
- 汇总结果

这让系统具备：

- 更强的可扩展性
- 更清晰的边界
- 更好的可审计性

### 6.4 面试表达

> 我把医院内部能力抽象成 MCP 风格工具层，例如身份验证、病例查询、诊疗记录查询等。Agent 不直接硬编码业务逻辑，而是通过规划和工具路由完成任务，这样工具层和推理层解耦，后续扩展新能力时只需要新增工具，不需要重写主链路。

## 7. 多模态交互能力

### 7.1 图片理解

接入入口：

- `POST /api/agent/query`

当前 `query` 支持图片输入。

支持方式包括：

- `image_url`
- `image_base64`

如果传的是本地路径，服务端会转成 Base64 再发给模型。

图片会和文本一起进入多模态模型。

### 7.2 语音播报

接入入口：

- `POST /api/agent/query`

当前 `query` 支持回答后自动做语音播报。

流程：

1. 文本回答生成完毕
2. 调用 Qwen / DashScope TTS
3. 生成音频文件
4. 返回本地路径和下载链接

不会再直接把整段音频 Base64 回给前端。

### 7.3 多模态信息如何进入短期记忆

短期记忆里不保存原始大文件，而是保存多模态摘要。

#### 用户侧

保存图片摘要：

- `mime_type`
- `image_url`
- 是否带 Base64

#### 助手侧

保存语音摘要：

- `speech_mime_type`
- `speech_model`
- `speech_voice`
- `speech_file_path`
- `speech_download_url`

这样既能保留多模态上下文，又避免数据过大和隐私风险。

### 7.4 面试表达

> 我给 Agent 接入了图片理解和语音播报两类多模态能力。图片作为输入进入多模态问答链路，语音作为输出增强可达性。同时多模态信息不会直接以原始二进制落短期记忆，而是沉淀成可追踪的摘要元数据，兼顾上下文连续性和存储成本。

## 8. Planner 设计

### 8.1 为什么要拆成 Planner / System / Finalizer

这三个 Prompt 分工不同：

- `Planner`
  - 先做任务拆解和执行路径规划
- `System / Controller`
  - 控制工具调用、权限校验和执行规则
- `Finalizer`
  - 基于已有证据整理最终答案

### 8.2 各自职责

#### Planner

解决：

- 先做什么
- 是否需要验证身份
- 是否需要查病例或就诊记录
- 是否需要结合图片

#### System / Controller

解决：

- 按什么规则调用工具
- 什么情况下停止
- 什么情况下继续查

#### Finalizer

解决：

- 如何只基于证据收敛最终回答
- 如何不暴露内部推理链

### 8.3 为什么不能合并成一个 Prompt

如果把三者混在一起，容易出现：

- 还没查完就急着回答
- 执行阶段和总结阶段目标冲突
- 工具调用顺序混乱
- 难以调试和审计

拆开之后更像一个标准控制流：

1. Planner 决定路线
2. Controller 执行路线
3. Finalizer 汇报结果

### 8.4 ReAct、CoT、自一致性怎么落地

当前 Planner 不是单轮静态规划，而是：

- 先用 Planner Prompt 产多个候选计划
- 通过不同温度做多次采样
- 再做自一致性合并

执行阶段按 ReAct 方式：

- 观察问题
- 决定动作
- 调工具
- 获得结果
- 再决定下一步

最终再由 Finalizer 做收敛。

### 8.5 面试表达

> 我把复杂问答链路拆成 Planner、Controller 和 Finalizer 三段。Planner 负责任务分解与路径选择，Controller 负责工具调用和执行约束，Finalizer 负责基于证据生成最终回答。这样能显著降低工具执行和答案生成的目标冲突，提升复杂任务下的稳定性和可控性。

## 9. 插拔式 API / 模块化能力

### 9.1 什么叫插拔式

这里的“插拔式”不是狭义插件市场，而是模块边界清晰，能力可以替换。

当前已经具备这些插拔点：

- LLM 客户端
- TTS 客户端
- 向量检索实现
- 记忆服务
- Tool / MCP 工具层

### 9.2 已经具备的模块边界

#### LLM 层

- `QwenClient`
- `QwenMCPAgent`

#### 语音层

- `QwenSpeechClient`

#### 记忆层

- `memory_service`
- `conversation_memory_service`
- `memory_preference_service`
- `memory_vector_service`

#### API 层

- `routes.py`

### 9.3 为什么这能支撑插拔

因为：

- 业务逻辑不直接耦合到某一个模型 SDK
- 记忆检索单独抽象成 service
- 工具层和主 Agent 层分离
- API 层只做编排，不承载核心算法

这让后续替换这些能力时成本更低，例如：

- 把 Qwen 换成别的模型
- 把 FAISS 换成别的向量索引
- 把语音播报换成别的 TTS

### 9.4 面试表达

> 我在工程上采用模块化与可替换设计，把 LLM、多模态、记忆、工具服务都收敛到独立 service 层，API 只负责编排。这样做的价值是降低模型和基础设施的耦合度，为后续更换向量引擎、语音服务或扩展工具能力提供稳定边界。

## 10. Query 的完整执行流程

对应 API：

- `POST /api/agent/query`

这是最核心的面试内容。

`POST /api/agent/query` 的全过程可以拆成 8 步。

### 步骤 1：接收请求

请求支持：

- `query`
- `images`
- `debug_planner`
- `enable_speech`
- `speech_voice`
- `speech_format`

服务初始化：

- `QwenClient`
- 可选的 `QwenSpeechClient`

这一阶段做的事情本质上是“准备运行环境”。

具体来说：

1. API 先接收用户自然语言问题
2. 如果用户同时上传了图片，也会把图片信息一起接收进来
3. 如果用户要求语音播报，会同步初始化语音客户端
4. 如果 `QWEN_API_KEY`、模型配置或者语音配置有问题，会在这一阶段直接报错，不会继续走后面的 Agent 流程

这样设计的原因是：

- 把基础依赖检查放到最前面
- 避免 Agent 已经跑了一半，最后才发现模型或语音服务不可用

面试时可以强调一点：

> Query 入口不是拿到请求就直接调模型，而是先做能力初始化和配置校验，确保主链路运行时不会因为底层依赖缺失而中途崩掉。

### 步骤 2：预识别患者

在正式问答前，系统会尝试先识别当前是哪个患者。

识别方式包括：

- 从问题里提取 `P0003`
- 从问题里提取手机号
- 如果后面工具调用已经返回了患者信息，也会再二次确认

这样做的目的，是为了提前把记忆上下文加载给 Agent。

这一阶段非常关键，因为记忆模块是按患者维度组织的。

如果系统在问答之前就能识别出患者是谁，那么后面的回答就不再是“纯即时问答”，而是“带上下文的个性化问答”。

预识别的逻辑是：

1. 先从本次 `query` 文本里尝试提取患者编码
2. 如果没有患者编码，再尝试提取手机号
3. 如果这一步仍然无法确认，就先按“无患者上下文”继续执行
4. 等工具调用真正拿到患者信息后，再做一次二次识别，用于本轮结束后的患者归属确认和短期记忆回写

为什么要分成“预识别”和“二次确认”两步：

- 预识别的目的是尽可能早地加载记忆
- 二次确认的目的是提高准确性，避免只靠正则识别带来的歧义

这里要注意一个实现边界：

- 预识别成功时，会在 `agent.run(...)` 前注入记忆上下文
- 二次确认成功时，不会在同一轮里重新构造一次记忆上下文
- 二次确认主要服务于本轮问答结束后的患者归属确认和短期记忆写回

这一步的本质不是完成身份验证，而是做“上下文绑定”。

也就是说：

- 这里解决的是“这个问题更可能对应哪个患者”
- 不是“是否已经通过隐私校验”

面试时这点很重要，因为很多系统会把“患者定位”和“身份验证”混在一起，但它们其实是两件事。

### 步骤 3：构造记忆上下文

如果识别到患者，会先读取三类记忆：

1. 最近 6 条短期记忆
2. 长期用户画像
3. 相关关键事件

其中关键事件是通过混合检索获得的，不是简单取最新几条。

这里是记忆模块真正介入 Query 的地方。

#### 3.1 短期记忆读取

系统会读取该患者最近几条短期对话，一般是最近 6 条。

目的有两个：

- 让模型知道最近几轮刚聊过什么，避免重复解释
- 保持多轮会话的连续性

例如用户刚刚问过：

- 最近一次就诊记录
- 用药建议
- 某张图片是不是相关材料

那这一轮回答前，模型就能先看到这些历史上下文。

#### 3.2 长期用户画像读取

系统还会读取该患者的长期画像，例如：

- 偏好什么样的表达方式
- 更关注哪些疾病或主题
- 是否喜欢简短回答
- 是否希望使用通俗语言

这一步的作用不是补业务事实，而是补“回答风格”和“用户偏好”。

也就是说：

- 短期记忆更偏上下文连续性
- 用户画像更偏个性化表达

#### 3.3 长期关键事件读取

关键事件的读取是第三块，也是最贴近业务事实的一块。

这里取的不是所有事件，也不是简单取最近几条，而是：

1. 用当前问题作为检索 query
2. 在该患者的长期关键事件里做混合检索
3. 返回和本次问题最相关的 topN 事件

这样做的原因是：

- 用户当前问“医生是谁”，相关事件应该更偏就诊记录
- 用户当前问“高脂血症”，相关事件应该更偏病例诊断
- 用户当前问“最近一次复诊”，相关事件应该更偏最新时间线

所以这一步本质上是在做：

> 先把长期记忆从“存量事实”变成“和当前问题相关的证据上下文”。

### 步骤 4：把记忆上下文注入 Agent

这些记忆不会直接暴露给用户，而是转成系统上下文消息注入模型。

这样模型在真正规划和回答前，就已经知道：

- 用户最近几轮聊过什么
- 这个用户偏好什么表达风格
- 和本次问题最相关的历史关键事件是什么

这一步是“记忆从存储层进入推理层”的关键桥梁。

系统不会把这些记忆当作最终答案直接返回，而是会把它们整理成结构化上下文，作为额外的模型输入。

注入之后，模型会同时看到三类信息：

1. 当前这次用户问题
2. 历史短期对话
3. 长期画像和关键事件

这样模型在真正开始规划时，就不是从零开始理解问题，而是带着“历史认知”来做决策。

这一层的价值主要有三个：

- 降低重复提问和重复回答
- 让输出风格更稳定
- 让工具调用更贴近历史业务事实

例如：

- 如果画像里写明用户喜欢简短、通俗的回答，Finalizer 就更容易生成符合偏好的回复
- 如果关键事件里已经有最近一次复诊记录，Planner 就可能减少无意义查询

面试时可以这样理解：

> 记忆注入不是为了让模型“背答案”，而是为了让它在规划和推理时拥有更完整的先验上下文。

### 步骤 5：执行 Planner + 工具调用

Agent 开始正式执行：

1. Planner 生成候选计划
2. 自一致性合并计划
3. Controller 按 ReAct 方式调用工具
4. 收集结构化证据
5. Finalizer 整理答案

这是整个 Query 里最核心的执行链路。

#### 5.1 Planner 生成候选计划

Planner 会根据：

- 当前用户问题
- 注入的记忆上下文
- 工具能力范围

先判断这类问题应该怎么拆解。

Planner 关注的问题一般包括：

- 需不需要先验证身份
- 是先查病例还是先查就诊记录
- 问题是否和图片有关
- 需要哪些工具
- 最终答案应该覆盖哪些重点

#### 5.2 自一致性合并计划

Planner 不只生成一个计划，而是会多次采样生成多个候选计划，再做一致性合并。

这么做的目的是：

- 降低单次采样带来的偶然性
- 提高多步任务的执行稳定性
- 避免工具顺序乱跳

这里的工程思路其实就是把“规划”从单次生成，升级成带投票和合并的 planning 过程。

#### 5.3 Controller 按 ReAct 方式执行

Controller 拿到计划之后，会按 ReAct 方式推进：

1. 先观察当前问题和上下文
2. 决定下一步调用哪个工具
3. 获取工具结果
4. 再决定是继续查还是停止

这一步强调的不是“自由生成”，而是“受控执行”。

也就是说，模型不是想答什么就答什么，而是被约束成：

- 先拿证据
- 再组织答案

#### 5.4 收集结构化证据

工具调用结果会被结构化保留，例如：

- 验证结果
- 患者信息
- 病例列表
- 就诊记录

这样后面的 Finalizer 就不是凭感觉总结，而是基于明确证据收敛答案。

#### 5.5 Finalizer 整理最终答案

Finalizer 的作用是：

- 只基于已拿到的证据生成用户可读答案
- 不暴露过长的内部思维链
- 在需要时说明证据不足或无法确认

这一步相当于把“内部执行轨迹”翻译成“用户可理解的业务答复”。

面试时建议你强调一点：

> Query 主链路不是让模型直接从问题跳到答案，而是先让 Planner 规划，再让 Controller 调工具拿证据，最后由 Finalizer 基于证据收敛答案，这样可控性和稳定性更强。

### 步骤 6：可选的语音播报

如果 `enable_speech = true`：

- 用语音合成模型把最终答案转成音频
- 保存到本地文件
- 返回下载链接

这一步是一个完全独立的后处理阶段。

它的特点是：

- 不影响文本问答主链路
- 只在文本答案已经生成后执行
- 失败时也不会反向污染主推理流程

也就是说，系统的主输出永远是文本答案，语音是附加增强能力。

具体流程是：

1. Finalizer 生成文本答案
2. 如果用户开启 `enable_speech`
3. 调用 TTS 模型生成音频
4. 把音频落本地文件
5. 返回 `speech_file_path` 和 `speech_download_url`

这一步的设计价值是：

- 前后链路解耦
- 多模态输出能力可以按需开启
- 不会让 TTS 问题影响问答本身

### 步骤 7：自动回写短期记忆

问答完成后，如果能识别患者，就自动写两条短期记忆：

- 用户问题
- 助手回答

如果本次有图片或语音，也会把多模态摘要一并写进去。

这是 Query 从“读取记忆”转向“生产记忆”的关键步骤。

具体来说：

#### 7.1 写入用户侧短期记忆

会记录：

- 用户原始问题
- 角色：`user`
- 本次图片摘要（如果有）

#### 7.2 写入助手侧短期记忆

会记录：

- 助手最终答案
- 角色：`assistant`
- 本次语音摘要（如果有）

这样做以后，下一轮 Query 在读取短期记忆时，就能把本轮内容也纳入上下文。

这里设计成“自动回写”的好处是：

- 前端不需要额外调用短期记忆接口
- 记忆链路天然闭环
- 用户每轮问答都会自然沉淀为下一轮的上下文

这一步本质上是：

> Query 不只是一个消费接口，它本身还是短期记忆的生产入口。

### 步骤 8：检查是否触发长期记忆沉淀

短期记忆写入后，会立刻做计数。

如果累计达到 5 轮阈值：

- 自动取最近 10 条短期对话
- 提炼长期用户画像
- 提炼对话类关键事件
- 更新长期记忆

所以 query 不只是“消费记忆”，它还承担了“短期记忆回写”和“长期记忆沉淀触发”的职责。

这是 Query 和长期记忆之间的闭环设计。

系统并不是每一轮都立即做长期抽取，而是：

1. 先把本轮问答写入短期记忆
2. 对该患者当前短期记忆总数做一次统计
3. 如果达到阈值，就触发长期沉淀

为什么不每轮都沉淀：

- 成本高
- 很多对话内容不稳定
- 容易把短期噪声直接写进长期记忆

为什么选择 5 轮作为阈值：

- 太短，信息不够稳定
- 太长，用户画像和对话事件又沉淀得太慢

所以当前版本选择 5 轮作为折中点。

触发后会发生两件事：

#### 8.1 更新长期用户画像

系统会基于最近几轮对话和已有偏好，重新生成：

- 用户关注主题
- 沟通风格
- 稳定偏好
- 画像摘要

#### 8.2 更新长期对话事件

系统会从最近几轮短期对话中抽取有医学价值的事件线索，例如：

- 提到某种疾病
- 提到某类症状
- 提到某类复诊关注点

这些内容会作为 `conversation` 来源的长期关键事件写入 `memory_events`。

这一阶段的本质是：

> 短期记忆是原材料，长期记忆是沉淀结果，Query 通过阈值触发把二者串成闭环。

### 一句话总结 Query 流程

> Query 的主链路是：先识别患者并加载短期记忆、用户画像和关键事件作为上下文，再由 Planner 驱动工具调用完成问答，最后把本轮问答写回短期记忆，并在达到阈值时自动沉淀成长记忆。

## 11. 这个项目当前最值得强调的工程点

### 11.1 不是简单聊天机器人

这个项目不是纯 prompt 问答，而是：

- 带工具调用
- 带记忆
- 带多模态
- 带规划
- 带长期沉淀

### 11.2 记忆架构是分层的

短期和长期记忆的来源、用途、更新频率都不同，所以分开设计。

### 11.3 Query 不是只读

它既消费记忆，也回写短期记忆，并能触发长期记忆沉淀。

### 11.4 混合检索是当前长期事件召回的核心

不是纯关键词，也不是纯向量，而是两者结合，并把来源显式标记出来。

### 11.5 工程上保留了可替换边界

模型、向量检索、多模态、工具服务都不是写死在一个大函数里。

## 12. 面试时最推荐的 5 个回答模板

### 12.1 记忆架构怎么设计

> 我把记忆分成短期记忆和长期记忆。短期记忆保存最近几轮会话内容，保证多轮问答连续性；长期记忆保存用户画像、关键事件和长期偏好，用于跨会话个性化回答。短期记忆会按阈值触发沉淀，把有长期价值的信息提炼成长期记忆。

### 12.2 为什么要把业务数据抽取和对话抽取拆开

> 因为业务数据和对话数据的更新频率完全不同。病例和诊疗记录更新频率低，适合增量或定时抽取；短期对话更新频率高，更适合按轮次阈值触发。如果合成一个统一接口，高频场景会重复扫描低频业务数据，成本和复杂度都不合理。

### 12.3 Planner 为什么要拆三段

> Planner 负责任务分解，Controller 负责工具执行约束，Finalizer 负责基于证据生成最终答案。拆开后能避免工具调用和答案生成目标冲突，复杂场景下的稳定性和可解释性都会更好。

### 12.4 Query 为什么是核心

> Query 是整个系统的主编排链路。它会先加载记忆上下文，再驱动 Planner 和工具调用生成答案，最后把本轮问答回写到短期记忆，并在达到阈值时自动沉淀为长期记忆。也就是说 Query 同时承担了上下文消费和记忆闭环两部分职责。

### 12.5 插拔式能力怎么体现

> 我把大模型、TTS、记忆服务、向量检索和工具层都抽成独立模块，API 层只负责编排。这样后续替换模型、替换向量引擎或者扩展工具能力时，只需要改对应模块，不需要重写主问答链路。

## 15. 接口白话说明

这一节专门给非科班复习使用。目标不是讲实现细节，而是讲清楚：

- 这个接口是干什么的
- 什么时候会用到
- 传什么最关键
- 返回结果大概是什么

### 15.1 Agent 接口

#### `POST /api/agent/query`

这是整个系统最核心的接口，也是前端最常调用的接口。

你可以把它理解成：

> “把用户的问题交给智能助手，智能助手会自己决定要不要查资料、查病例、查就诊记录、结合图片分析，然后返回最终答案。”

这个接口支持的能力最多：

- 普通文本问答
- 结合图片的多模态问答
- 语音播报
- 自动读取记忆上下文
- 自动写入短期记忆
- 自动在达到阈值时触发长期记忆更新

最关键的输入字段：

- `query`
  - 用户真正的问题
- `images`
  - 可选，用户上传的图片
- `enable_speech`
  - 是否要把答案额外生成语音
- `debug_planner`
  - 是否返回 Planner 调试信息

最关键的输出字段：

- `answer`
  - 最终文字答案
- `tool_outputs`
  - 工具调用结果
- `speech_file_path`
  - 语音文件本地路径
- `speech_download_url`
  - 语音下载地址

什么时候用：

- 用户提问时
- 用户带图片提问时
- 用户想听语音播报时

### 15.2 记忆接口

#### `GET /api/memory/preferences`

这个接口是“查询长期偏好配置”的。

你可以理解成：

> “看看这个患者之前有没有设置过自己喜欢怎样被回答。”

适合查看的内容包括：

- 偏好称呼
- 回答风格
- 回答长度
- 语言
- 关注主题

#### `PUT /api/memory/preferences`

这个接口是“保存长期偏好配置”的。

你可以理解成：

> “把用户主动设置的个性化偏好保存起来，以后问答都可以参考。”

什么时候用：

- 用户在设置页里修改偏好时
- 运营或客服帮用户维护偏好时

#### `POST /api/memory/conversations`

这个接口是“手动写入短期记忆”的。

系统平时已经会自动写，但这个接口依然保留，主要作用是：

- 测试
- 补数据
- 联调

你可以把它理解成：

> “强行往短期记忆库里塞一条对话。”

#### `GET /api/memory/conversations`

这个接口是“查看短期记忆”的。

它适合做两件事：

- 确认问答后有没有把短期记忆写进去
- 看最近几轮对话到底保存了什么

如果你在调试“为什么这次回答带了历史上下文”，这个接口很有用。

#### `POST /api/memory/extract/business`

这个接口是“从业务数据提炼长期关键事件”的。

它会扫描：

- 病例
- 就诊记录

然后提炼出长期关键事件，例如：

- 某次病例诊断
- 某次就诊
- 某次复诊

你可以把它理解成：

> “把原始业务数据整理成更适合记忆和检索的时间线事件。”

这个接口的输出结果会进入：

- `memory_events`
- 如果向量能力可用，还会进入向量索引

#### `POST /api/memory/extract/conversation`

这个接口是“从短期对话提炼长期记忆”的。

它不是让你自己传对话文本，而是：

- 根据 `patient_id`
- 去短期记忆库里取最近 N 条对话
- 再提炼出用户画像和对话事件

你可以理解成：

> “把最近几轮对话里有长期价值的信息沉淀下来。”

它主要沉淀两类东西：

- 用户画像
- 对话类关键事件

#### `GET /api/memory/events`

这个接口是“查看已经沉淀好的长期关键事件”。

如果你想知道一个患者当前系统里都有哪些长期事件，这个接口最直接。

这些事件可能来自：

- 业务数据
- 短期对话抽取

#### `POST /api/memory/search/events`

这个接口是“检索长期关键事件”的。

和 `GET /api/memory/events` 的区别是：

- `GET /api/memory/events` 是全量看已有事件
- `POST /api/memory/search/events` 是围绕当前问题去找最相关的事件

这个接口会做：

- 关键词匹配
- 在向量依赖和 embedding 配置可用时做向量检索
- 混合打分

返回里你还能看到：

- 是关键词命中的
- 是向量命中的
- 还是两者都命中的

如果当前环境里 FAISS 不可用，或者 Qwen embedding 没法正常调用，这个接口会退化成非向量路径。

这个接口非常适合调试“长期记忆为什么被召回”。

#### `GET /api/memory/profile`

这个接口是“查看长期用户画像”。

它返回的是：

- 画像摘要
- 沟通风格
- 偏好主题
- 稳定偏好
- 画像来源

你可以理解成：

> “系统对这个用户形成了怎样的长期理解。”

### 15.3 患者基础信息接口

#### `POST /api/patients`

创建患者。

适合在系统初始化、录入测试数据或新增真实患者资料时使用。

#### `GET /api/patients`

查询患者列表。

适合后台查看现有患者，或者做测试时先看看系统里都有哪些患者。

#### `GET /api/patients/{patient_id}`

查看某个患者的详细基础信息。

#### `PUT /api/patients/{patient_id}`

更新患者基础信息。

例如：

- 修改手机号
- 修改地址
- 更新紧急联系人

### 15.4 病例接口

#### `POST /api/medical-cases`

创建病例。

病例偏重“诊断和医学描述”，一般包含：

- 诊断
- 主诉
- 现病史
- 既往史
- 治疗方案

#### `GET /api/medical-cases`

查询病例列表。

可以按 `patient_id` 过滤，只看某个患者的病例。

#### `GET /api/medical-cases/{case_id}`

查看单条病例详情。

#### `PUT /api/medical-cases/{case_id}`

更新病例内容。

适合补充诊断、完善治疗方案或修改病例描述。

### 15.5 就诊记录接口

#### `POST /api/visit-records`

创建就诊记录。

就诊记录偏重“某一次具体就诊行为”，通常包含：

- 就诊时间
- 科室
- 医生
- 摘要
- 备注

#### `GET /api/visit-records`

查询就诊记录列表。

可以按 `patient_id` 过滤，只看某个患者的就诊记录。

#### `GET /api/visit-records/{visit_record_id}`

查看单条就诊记录详情。

#### `PUT /api/visit-records/{visit_record_id}`

更新某条就诊记录。

适合补充医生、备注、就诊摘要等。

### 15.6 最后怎么理解这些接口的关系

如果你把所有接口看成一个整体，可以这样记：

- `/api/agent/query`
  - 真正给用户用的主问答接口
- `/api/memory/*`
  - 给系统做记忆沉淀、读取、检索、配置用的接口
- `/api/patients/*`
  - 管患者基础资料
- `/api/medical-cases/*`
  - 管病例
- `/api/visit-records/*`
  - 管就诊记录

所以最简单的理解是：

> 患者、病例、就诊记录是原始业务数据；记忆接口负责把这些数据和对话沉淀成长期记忆；`/api/agent/query` 负责在问答时把这些能力都串起来。
