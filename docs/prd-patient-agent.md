# 患者智能辅助 Agent PRD

## 1. 文档信息

- 产品名称：患者智能辅助 Agent
- 文档版本：V1.0
- 文档日期：2026-03-15
- 文档状态：初稿

## 2. 项目概述

患者智能辅助 Agent 面向医院就诊患者，提供诊前咨询、报告解读、诊后答疑、就诊记录查询和语音辅助等能力。产品以大模型、知识检索、业务工具调用和用户记忆体系为基础，为患者提供连续、可理解、可追溯的智能服务，同时降低医生处理基础重复咨询的工作负担。

## 3. 项目目标

### 3.1 业务目标

1. 为患者提供 7x24 小时智能咨询服务。
2. 缩短诊前咨询平均响应时间，提升问题首答率。
3. 降低患者对检查报告、检验结果、就诊流程的理解门槛。
4. 承接诊后常见问题，减少医生基础重复答疑耗时。
5. 通过智能问答与工具调度，将医生处理基础咨询时间减少 40%。

### 3.2 产品目标

1. 打通患者咨询、医院知识库和医院内部业务系统。
2. 建立短期会话记忆和长期患者画像体系。
3. 提供文本、语音、图片等多模态交互能力。
4. 形成可复用、可配置、可跨院区部署的智能 Agent 平台。

## 4. 业务背景

当前患者就诊服务存在以下问题：

1. 诊前咨询量大，人工客服和医生无法及时响应。
2. 检查报告与检验结果专业性较强，患者难以理解。
3. 诊后康复、复诊、用药相关问题频繁，医护人员重复答疑成本高。
4. 医院内部信息系统分散，患者问题往往需要跨系统查询与解释。
5. 老年患者或文化程度较低患者在文字理解和信息获取方面存在障碍。

## 5. 产品定位

本产品不是单纯聊天机器人，而是面向患者服务场景的智能 Agent 平台。其核心能力包括：

1. 基于知识库和患者历史数据进行上下文理解与问答。
2. 调用院内业务工具完成认证、查询、读取与任务协同。
3. 结合用户短期会话和长期画像实现连续个性化服务。
4. 在高风险医疗场景下提供保守回答、风险提示和人工升级能力。

## 6. 用户角色

### 6.1 主要用户

1. 患者本人
2. 患者家属或陪诊人

### 6.2 次要用户

1. 医生
2. 护士
3. 导诊或客服人员
4. 医院运营管理人员
5. 信息化与技术团队

## 7. 使用场景

1. 诊前咨询：挂号准备、就诊流程、检查前注意事项、科室选择。
2. 报告解读：检验单、检查报告、异常指标说明。
3. 诊后答疑：康复问题、复诊提醒、常见不适说明。
4. 个人记录查询：病历、就诊记录、检查结果、历史诊疗摘要。
5. 多模态辅助：语音播报、图片上传、医学图像辅助理解。

## 8. 范围定义

### 8.1 MVP 范围

1. 文本问答
2. 基于知识库的 RAG 检索回答
3. 患者身份认证
4. 本人病历和就诊记录查询
5. 检验检查报告文本解读
6. 短期记忆和长期画像记忆
7. Agent 自动调用业务工具
8. 语音播报

### 8.2 二期范围

1. 医学图像辅助理解
2. 家属授权访问
3. 多院区部署与配置迁移
4. 个性化回答风格配置
5. 诊后随访任务编排
6. 更复杂的多步骤任务执行

### 8.3 非本期范围

1. 替代医生进行正式诊断
2. 独立生成处方或调整用药方案
3. 自动完成收费、退款等复杂交易闭环

## 9. 成功指标

### 9.1 核心业务指标

1. 医生基础重复咨询耗时下降 40%。
2. 患者智能咨询首答覆盖率大于 80%。
3. 患者满意度大于 85%。

### 9.2 产品指标

1. 普通问答首答时延小于 3 秒。
2. 工具型查询成功率大于 95%。
3. 报告解读可用率大于 90%。
4. 高风险问题转人工或提示就医覆盖率达到 100%。
5. 关键回答引用或溯源覆盖率达到 100%。

## 10. 产品原则

1. 医疗安全优先，严禁越权回答。
2. 只有在有依据时才输出明确说明。
3. 涉及隐私数据时必须先认证后查询。
4. 所有关键回答必须可追溯。
5. 回答内容必须区分医学信息说明与诊疗建议。

## 11. 功能架构

产品功能由以下模块组成：

1. 智能问答中心
2. 患者身份认证与权限控制
3. 病历与就诊记录查询
4. 报告解读
5. 多模态交互
6. 记忆系统
7. Agent 工具调度
8. 配置化与多租户支持

## 12. 详细功能需求

### 12.1 智能问答中心

#### 12.1.1 功能描述

支持患者通过文本或语音向系统提问，系统识别用户意图后，结合知识库、用户上下文、业务数据与工具结果输出回答。

#### 12.1.2 用户故事

1. 作为患者，我希望快速得到诊前准备信息，以减少来回咨询。
2. 作为患者，我希望系统可以记住我刚才问过的问题，避免重复描述。
3. 作为患者，我希望系统在回答时给出依据，便于判断可靠性。

#### 12.1.3 功能点

1. 支持文本输入和语音输入。
2. 支持连续多轮对话。
3. 支持上下文继承和意图识别。
4. 支持展示回答依据和来源。
5. 支持不确定性提示和人工升级建议。

#### 12.1.4 输入

1. 用户问题
2. 会话上下文
3. 用户认证状态
4. 用户历史画像
5. 附件信息

#### 12.1.5 输出

1. 回答内容
2. 风险提示
3. 来源引用
4. 建议动作

#### 12.1.6 业务规则

1. 急症、诊断、停药、加药等高风险问题不得直接给出确定性医疗决策。
2. 在无可靠依据时，不得输出强结论。
3. 低置信度情况下需要降级为保守回答或推荐人工咨询。

### 12.2 患者身份认证与权限控制

#### 12.2.1 功能描述

患者在访问个人隐私信息前需要完成身份认证，系统根据权限范围决定可查询的数据类型。

#### 12.2.2 功能点

1. 手机号认证
2. 证件号或就诊卡信息认证
3. 短信验证码校验
4. 会话级权限控制
5. 查询行为审计

#### 12.2.3 业务规则

1. 未认证用户仅可访问通用知识问答。
2. 已认证用户仅可访问本人信息。
3. 敏感字段需按规则脱敏展示。
4. 二期支持家属授权代查机制。

### 12.3 病历与就诊记录查询

#### 12.3.1 功能描述

支持患者查询门诊、住院、检验、检查等历史记录，并以患者友好方式展示摘要。

#### 12.3.2 功能点

1. 查询最近就诊记录
2. 查询检验检查单
3. 输出关键事件摘要
4. 支持按时间筛选
5. 支持卡片化展示

#### 12.3.3 业务规则

1. 查询行为必须建立在认证基础上。
2. 返回内容需要做患者语言转换。
3. 原始报告字段可在详细信息中查看。

### 12.4 报告解读

#### 12.4.1 功能描述

支持对检验检查报告进行通俗化解释，帮助患者理解异常指标和可能含义。

#### 12.4.2 功能点

1. 报告摘要生成
2. 异常项高亮
3. 异常指标解释
4. 风险级别提示
5. 就医建议说明

#### 12.4.3 业务规则

1. 不输出正式确诊结论。
2. 不替代医生解读。
3. 遇到高危指标时优先提示尽快就医。
4. 所有解释应标注为一般性说明。

### 12.5 多模态交互

#### 12.5.1 功能描述

支持语音播报、语音识别、图片上传和医学图像辅助理解。

#### 12.5.2 功能点

1. 文本转语音播报
2. 语音输入转文本
3. 报告图片上传
4. OCR 文本提取
5. 医学图像辅助理解（二期）

#### 12.5.3 业务规则

1. 医学图像理解仅作为辅助说明。
2. 图像结果必须提示不能替代医生阅片。

### 12.6 记忆系统

#### 12.6.1 功能描述

建立短期会话记忆和长期患者画像，用于提升连续对话体验和个性化服务效果。

#### 12.6.2 功能点

1. 短期记忆缓存
2. 长期画像抽取
3. 关键事件抽取
4. 用户偏好记录
5. 记忆刷新与更新
6. 用户记忆查看与清理

#### 12.6.3 业务规则

1. 长期记忆只保存筛选后的结构化信息和摘要，不直接保存全部原始对话。
2. 高敏信息需要加密存储。
3. 记忆需支持过期和纠错机制。

### 12.7 Agent 工具调度

#### 12.7.1 功能描述

将病例查询、记录调取、身份认证等业务能力封装为 MCP Server，由 Agent 根据意图自动调用。

#### 12.7.2 功能点

1. 身份认证工具
2. 病历查询工具
3. 就诊记录查询工具
4. 报告查询工具
5. 知识库检索工具
6. 转人工工具

#### 12.7.3 业务规则

1. 仅在必要时调用工具。
2. 工具调用必须记录日志和 trace。
3. 工具失败时需有降级或兜底策略。

### 12.8 配置化与多租户支持

#### 12.8.1 功能描述

通过配置化和插件化方式支持多院区、多科室、多版本接入。

#### 12.8.2 功能点

1. 医院级知识库配置
2. 科室级 Prompt 配置
3. 工具启停配置
4. 回答风格配置
5. 跨院区配置迁移

## 13. 核心用户流程

### 13.1 诊前咨询流程

1. 患者进入首页。
2. 输入“做胃镜前能不能喝水”。
3. 系统识别为检查前注意事项。
4. 系统检索医院知识库。
5. 返回标准说明、注意事项和确认建议。

### 13.2 报告解读流程

1. 患者选择历史报告或上传报告图片。
2. 系统识别报告类型并提取内容。
3. 系统结合报告文本和知识库进行解读。
4. 返回摘要、异常项说明和风险提示。

### 13.3 个人记录查询流程

1. 患者完成身份认证。
2. 输入“我上次 CT 结果是什么”。
3. 系统识别需要访问个人隐私数据。
4. 系统调用记录查询工具。
5. 返回检查时间、结果摘要和报告入口。

### 13.4 诊后答疑流程

1. 患者输入“术后两周还有轻微疼痛正常吗”。
2. 系统结合术后知识和患者近期病历摘要进行分析。
3. 返回一般性说明和观察建议。
4. 如命中风险规则，提示尽快联系医生或到院复诊。

## 14. 页面与交互要求

### 14.1 首页

1. 展示文本输入框、语音入口和快捷问题。
2. 支持查看常见场景入口，如诊前咨询、报告解读、我的记录。

### 14.2 对话页

1. 展示用户消息和系统回复。
2. 展示来源引用、风险提示、后续建议。
3. 对高风险问题进行红色告警显示。

### 14.3 报告解读页

1. 展示报告摘要。
2. 高亮异常指标。
3. 展示患者友好说明和风险提示。

### 14.4 记录查询页

1. 支持按时间和类型筛选。
2. 以卡片形式展示门诊、检查、检验记录摘要。

### 14.5 老年模式

1. 支持大字号。
2. 支持语音播报。
3. 支持更简洁的语言表达。

## 15. 数据需求

### 15.1 核心数据对象

1. 患者基础信息
2. 认证信息
3. 会话记录
4. 用户画像
5. 关键事件
6. 病例摘要
7. 就诊记录
8. 检验检查报告
9. 知识库文档
10. 工具调用日志
11. 审计日志

### 15.2 数据处理要求

1. 敏感信息必须加密存储。
2. 对话、调用、查询记录需可审计。
3. 长期记忆写入需经过规则筛选和结构化抽取。

## 16. 风控与合规要求

1. 禁止在没有依据时输出诊断、处方或停药建议。
2. 急症关键词必须触发风险告警。
3. 所有隐私数据读取行为必须可审计。
4. 输出内容需要带有医疗场景免责声明。
5. 模型回答前后均需经过安全规则检查。

## 17. 非功能需求

### 17.1 性能要求

1. 普通问答响应时间小于 3 秒。
2. 复杂查询响应时间小于 8 秒。

### 17.2 可用性要求

1. 核心服务可用性不低于 99.9%。
2. 工具服务异常时主流程可降级。

### 17.3 安全要求

1. 传输层加密。
2. 存储层敏感数据加密。
3. 支持角色权限控制。

### 17.4 扩展性要求

1. 新增工具不影响主链路。
2. 新增医院或院区通过配置接入。
3. 记忆模块和工具模块支持独立扩展。

## 18. 验收标准

1. 患者可完成身份认证并查询个人记录。
2. 系统可对检验检查报告进行文本解读。
3. 系统支持多轮连续问答。
4. Agent 可根据不同意图自动调用对应工具。
5. 记忆系统可保存并检索短期与长期信息。
6. 回答可展示引用来源和风险提示。
7. 系统支持语音播报。
8. 高风险问题可稳定触发人工升级或就医提示。

## 20. 风险与待确认项

1. 医院内部系统接入标准是否统一。
2. 患者身份认证方式是否已有院内标准。
3. 医学图像能力是否允许对接正式诊疗辅助场景。
4. 长期记忆的保留周期、用户可见性和删除机制是否有明确合规要求。
5. 医院是否已有统一知识库和内容审核流程。
